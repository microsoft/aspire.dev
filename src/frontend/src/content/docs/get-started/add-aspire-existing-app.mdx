Add Aspire to the app you already have instead of rebuilding your solution around a new template. Start by choosing the AppHost style that fits your repo, then register the apps and infrastructure you already run today.

:::note[Definition]{icon="approve-check-circle"}
**Aspireify** (verb): To transform an existing application into a distributed, observable, and orchestrated system by adding Aspire.
:::

## Why add Aspire to an existing app?

As distributed applications grow, local development often turns into a collection of fragile scripts, copied connection strings, and startup-order tribal knowledge. Aspire gives you a single orchestration layer for the apps and services you already own. Define the relationships once in code, and Aspire handles service discovery, configuration injection, startup ordering, and dashboard visibility.

You can also adopt Aspire incrementally. Start with orchestration, add telemetry when you're ready, and bring in shared resources such as caches, databases, and queues as your app grows.

## Start with your scenario

This guide is organized around common starting points rather than around a preferred language.

- **Existing .NET solution**: You already have one or more `.csproj` projects and want Aspire to orchestrate them.
- **Existing JavaScript or TypeScript workspace**: You already have Node.js services, Vite apps, or a monorepo and want Aspire to coordinate them.
- **Mixed-language system**: Your system already spans multiple runtimes, such as a .NET API, Python worker, and JavaScript frontend.

In every scenario, you can choose either AppHost style.

## Choose your AppHost

The AppHost is the orchestration layer. Your choice here changes how you express orchestration, not what Aspire can orchestrate.

<Tabs syncKey='aspire-lang'>
<TabItem id='csharp' label='C#'>

Use a C# AppHost when your repo already centers on a .NET solution or when you prefer project references and typed resource definitions.

- Lives in an AppHost project with `AppHost.cs`
- Uses methods such as `AddProject<T>`, `AddNodeApp`, `AddPythonApp`, and `AddContainer`
- Fits naturally into existing .NET build, IDE, and solution workflows

</TabItem>
<TabItem id='typescript' label='TypeScript'>

Use a TypeScript AppHost when your repo already centers on a Node.js workspace or when you prefer path-based orchestration in TypeScript.

- Lives in `apphost.ts`
- Uses methods such as `addProject`, `addNodeApp`, `addPythonApp`, and `addContainer`
- Fits naturally into existing package-manager and monorepo workflows

</TabItem>
</Tabs>

:::tip[No runtime lock-in]
Both AppHost styles can orchestrate .NET, JavaScript, Python, containers, and other supported resource types. Pick the AppHost that best fits your repo and team, not the one that matches a single service language.
:::

### Scenario-specific requirements

**For .NET apps:**

- One or more `.csproj` projects
- A solution file is recommended for a C# AppHost, but not required for a TypeScript AppHost

**For JavaScript or TypeScript apps:**

- A Node.js app, Vite app, or monorepo workspace
- A package manager configuration such as `package.json`, `pnpm-workspace.yaml`, or equivalent

**For mixed-language apps:**

- Install each runtime your services require, such as Python, Node.js, and .NET

## Overview of the process

Adding Aspire to an existing app usually follows these steps:

<Steps>

1. **Choose an AppHost** that fits your repo and workflow.
2. **Initialize Aspire support** with `aspire init`.
3. **Register your existing apps** and shared resources in the AppHost.
4. **Add telemetry and integrations** where they add value.
5. **Run and verify** the full system with Aspire orchestration.

</Steps>

## Initialize Aspire support

The `aspire init` command creates the orchestration layer and helps wire the first set of resources into it.

<Tabs syncKey='aspire-lang'>
<TabItem id='csharp' label='C#'>

<Steps>

1. Navigate to the root of your existing solution or repo:

   ```bash title="Navigate to your solution or repo"
   cd /path/to/your-solution
   ```

2. Run `aspire init`:

   ```bash title="Initialize Aspire with a C# AppHost"
   aspire init
   ```

   The command runs in interactive mode by default. It can detect your existing projects, create an AppHost project, and add the initial hosting packages and references.

</Steps>

<LearnMore>
For more details on the `aspire init` command and its options, see the [CLI reference: `aspire init`](/reference/cli/commands/aspire-init/).
</LearnMore>

After initialization, a typical solution might look like this:

<FileTree>
    - ExampleEcommerce.sln (updated)
    - src/
        - **ExampleEcommerce.AppHost/** (new)
            - **ExampleEcommerce.AppHost.csproj**
            - **AppHost.cs**
        - Api/
            - ExampleEcommerce.Api.csproj
        - Web/
            - ExampleEcommerce.Web.csproj
        - OrderProcessor/
            - ExampleEcommerce.OrderProcessor.csproj
</FileTree>

Starter AppHost:

```csharp title="AppHost.cs — Initial state"
var builder = DistributedApplication.CreateBuilder(args);

// TODO: Add resources here

builder.Build().Run();
```

</TabItem>
<TabItem id='typescript' label='TypeScript'>

<Steps>

1. Navigate to the root of your existing workspace or repo:

   ```bash title="Navigate to your workspace or repo"
   cd /path/to/your-workspace
   ```

2. Run `aspire init` with the TypeScript AppHost option:

   ```bash title="Initialize Aspire with a TypeScript AppHost"
   aspire init --language typescript
   ```

   If you omit `--language typescript`, choose **TypeScript** when the CLI prompts for the AppHost language.

</Steps>

<LearnMore>
For more details on the `aspire init` command and its options, see the [CLI reference: `aspire init`](/reference/cli/commands/aspire-init/).
</LearnMore>

<FileTree>
    - my-store/

            - api/
                - package.json
                - src/
            - web/
                - package.json
                - src/
</FileTree>

Starter AppHost:

```typescript title="apphost.ts — Initial state"
import { createBuilder } from './.modules/aspire.js';

```

## Model your existing apps in the AppHost

Once you have an AppHost, register the resources you already run today. Each app, container, database, queue, or executable becomes a resource in the AppHost.

Two patterns stay the same across AppHost styles:

- Use `WithReference` or `withReference` to declare resource dependencies and enable service discovery.
- Use `WaitFor` or `waitFor` when one resource should wait for another to be ready.

### Scenario: Existing .NET solution

Use this approach when your existing app is already composed of one or more .NET projects.

<Tabs syncKey='aspire-lang'>
<TabItem id='csharp' label='C#'>

If you use a C# AppHost, `aspire init` will often add the initial project references for you. If you add more projects later, add references from the AppHost project so `AddProject<T>` can use the generated `Projects.*` marker types.

```bash title="Add more project references to the AppHost"
dotnet add reference ../Api/ExampleCommerce.Api.csproj
dotnet add reference ../Web/ExampleCommerce.Web.csproj
dotnet add reference ../OrderProcessor/ExampleCommerce.OrderProcessor.csproj
```

Then model the services in `AppHost.cs`:

```csharp title="AppHost.cs — Existing .NET solution"
var builder = DistributedApplication.CreateBuilder(args);

var api = builder.AddProject<Projects.ExampleCommerce_Api>("api")
    .WithHttpHealthCheck("/health");

var web = builder.AddProject<Projects.ExampleCommerce_Web>("web")
    .WithExternalHttpEndpoints()
    .WithReference(api)
    .WaitFor(api);

var worker = builder.AddProject<Projects.ExampleCommerce_OrderProcessor>("worker")
    .WithReference(api);

builder.Build().Run();
```

</TabItem>
<TabItem id='typescript' label='TypeScript'>

If you use a TypeScript AppHost, you can point directly to the existing `.csproj` files from `apphost.ts`.

```typescript title="apphost.ts — Existing .NET solution"
import { createBuilder } from './.modules/aspire.js';

const builder = await createBuilder();

const api = await builder
    .addProject('api', './src/Api/ExampleCommerce.Api.csproj')
    .withHttpHealthCheck({ path: '/health' });

await builder
    .addProject('web', './src/Web/ExampleCommerce.Web.csproj')
    .withExternalHttpEndpoints()
    .withReference(api)
    .waitFor(api);

await builder
    .addProject('worker', './src/OrderProcessor/ExampleCommerce.OrderProcessor.csproj')
    .withReference(api);

await builder.build().run();
```

</TabItem>
</Tabs>

### Scenario: Existing JavaScript or TypeScript workspace

Use this approach when your repo already contains Node.js services, Vite apps, or a JavaScript monorepo.

```csharp title="AppHost.cs — Existing JavaScript workspace"
    .WithReference(api)
    .WaitFor(api);

```

### Scenario: Mixed-language system

Use this approach when your system already spans multiple runtimes. The AppHost can model the shared infrastructure once and connect each service to it.

Add the hosting integrations you need for the service languages you are bringing in:

```bash title="Add language hosting support as needed"
aspire add javascript
aspire add python
```

<Tabs syncKey='aspire-lang'>
<TabItem id='csharp' label='C#'>

```csharp title="AppHost.cs — Mixed-language system"
var builder = DistributedApplication.CreateBuilder(args);

var cache = builder.AddRedis("cache");

var api = builder.AddProject<Projects.Store_Api>("api")
    .WithReference(cache)
    .WithHttpHealthCheck("/health");

var worker = builder.AddPythonApp("worker", "../worker", "main.py")
    .WithReference(cache);

var web = builder.AddViteApp("web", "../web")
    .WithExternalHttpEndpoints()
    .WithReference(api)
    .WaitFor(api);

builder.Build().Run();
```

</TabItem>
<TabItem id='typescript' label='TypeScript'>

```typescript title="apphost.ts — Mixed-language system"
import { createBuilder } from './.modules/aspire.js';

const builder = await createBuilder();

const cache = await builder.addRedis('cache');

const api = await builder
    .addProject('api', './src/Store.Api/Store.Api.csproj')
    .withReference(cache)
    .withHttpHealthCheck({ path: '/health' });

await builder
    .addPythonApp('worker', './worker', 'main.py')
    .withReference(cache);

await builder
    .addViteApp('web', './web')
    .withExternalHttpEndpoints()
    .withReference(api)
    .waitFor(api);

await builder.build().run();
```

## Add telemetry configuration (optional)

Telemetry is configured inside your services, not in the AppHost itself. The right setup depends on the service language you are bringing into Aspire.

### For .NET services

If your app includes .NET services, ServiceDefaults is the standard way to add observability, resilience, and health checks.

<Steps>

1. Add ServiceDefaults if you did not enable it during `aspire init`:

   ```bash title="Add ServiceDefaults"
   dotnet new aspire-servicedefaults -n YourProject.ServiceDefaults
   dotnet sln add YourProject.ServiceDefaults
   dotnet add YourProject reference YourProject.ServiceDefaults
   ```

2. Update your service's `Program.cs`:

   ```csharp title="Program.cs — Add ServiceDefaults"
   var builder = WebApplication.CreateBuilder(args);

   builder.AddServiceDefaults();

   var app = builder.Build();

   app.MapDefaultEndpoints();
   app.Run();
   ```

</Steps>

<LearnMore>
For more information, see [Service Defaults](/get-started/csharp-service-defaults/).
</LearnMore>

### For Node.js and TypeScript services

If your app includes Node.js services, configure OpenTelemetry inside the service and point it at the Aspire OTLP endpoint.

<Steps>

1. Install the OpenTelemetry packages:

   ```bash title="Install OpenTelemetry packages"
   npm install @opentelemetry/api @opentelemetry/sdk-node \
       @opentelemetry/auto-instrumentations-node \
       @opentelemetry/exporter-trace-otlp-grpc \
       @opentelemetry/exporter-metrics-otlp-grpc
   ```

2. Create a telemetry bootstrap file:

   ```typescript title="telemetry.ts"
   import { NodeSDK } from '@opentelemetry/sdk-node';
   import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';

   const sdk = new NodeSDK({
       instrumentations: [getNodeAutoInstrumentations()],
   });

   sdk.start();
   ```

3. Import it first in your app entry point:

   ```typescript title="src/server.ts"
   import './telemetry';

   import express from 'express';

   const app = express();
   ```

</Steps>

### For other runtimes

Use the OpenTelemetry SDK or instrumentation library that matches the runtime you are already using, then export telemetry to the OTLP endpoint Aspire provides during local orchestration.

## Add integrations (optional)

Aspire integrations simplify common infrastructure such as Redis, PostgreSQL, RabbitMQ, and more. When you add an integration to the AppHost and reference it from dependent apps, Aspire injects the configuration those apps need.

<Steps>

1. Identify the shared resources in your system, such as databases, caches, queues, or storage.
2. Add the integration to the AppHost.
3. Reference that resource from the apps that depend on it.
4. Configure the client library inside each consuming service.

</Steps>

For example, to add Redis:

```bash title="Add a Redis integration"
aspire add redis
```

<Tabs syncKey='aspire-lang'>
<TabItem id='csharp' label='C#'>

```csharp title="AppHost.cs — Share Redis across services"
var builder = DistributedApplication.CreateBuilder(args);

var cache = builder.AddRedis("cache");

var api = builder.AddProject<Projects.Api>("api")
    .WithReference(cache)
    .WithHttpHealthCheck("/health");

var worker = builder.AddNodeApp("worker", "../worker", "index.js")
    .WithNpm()
    .WithReference(cache);

builder.Build().Run();
```

</TabItem>
<TabItem id='typescript' label='TypeScript'>

```typescript title="apphost.ts — Share Redis across services"
import { createBuilder } from './.modules/aspire.js';

const builder = await createBuilder();

const cache = await builder.addRedis('cache');

const api = await builder
    .addProject('api', './src/Api/Api.csproj')
    .withReference(cache)
    .withHttpHealthCheck({ path: '/health' });

await builder
    .addNodeApp('worker', './worker', 'index.js')
    .withNpm()
    .withReference(cache);

await builder.build().run();
```

</TabItem>
</Tabs>

Example client configuration inside the consuming services:

```bash title="Add Redis client to a .NET project"
dotnet add YourApi package Aspire.StackExchange.Redis
```

```csharp title="Program.cs — Configure Redis in .NET"
var builder = WebApplication.CreateBuilder(args);
builder.AddRedisClient("cache");
```

```bash title="Add Redis client to a Node.js project"
npm install redis
```

```javascript title="Configure Redis in Node.js"
import { createClient } from 'redis';

const client = createClient({
    socket: {
        host: process.env.CACHE_HOST,
        port: Number(process.env.CACHE_PORT),
    },
});
```

## Run and verify

Once your AppHost models the system you want to run, start everything together with the Aspire CLI.

<Steps>

1. From the directory that contains your AppHost, run:

   ```bash title="Run your application with Aspire"
   aspire run
   ```

2. Wait for the CLI to discover the AppHost, launch the resources, and print the dashboard URL.

   ```bash title="Example output" mark={3}
   Finding apphosts...

   Dashboard: https://localhost:17068/login?t=example

   Press CTRL+C to stop the apphost and exit.
   ```

3. Open the dashboard in your browser and verify:

   - All resources start successfully
   - Service dependencies appear in the expected order
   - Logs, traces, and metrics are visible
   - Endpoints and environment variables look correct

4. Exercise the actual app flows you care about, such as frontend-to-API calls, worker jobs, or database access.

5. Stop the system by pressing <Kbd windows="Ctrl+C" mac="⌘+C" /> in your terminal.

</Steps>

:::tip
The [Aspire dashboard](/dashboard/explore/) is useful even before you add deeper telemetry or integrations. It gives you a single place to inspect startup order, logs, endpoints, and configuration.
:::

### Compare with Docker Compose

If you currently orchestrate the app with Docker Compose, the same relationships translate into an AppHost with less duplicated configuration.

<Tabs>
<TabItem label='Docker Compose'>

```yaml title="docker-compose.yml" showLineNumbers
services:
  postgres:
    image: postgres:latest
    environment:
      - POSTGRES_PASSWORD=postgres
      - POSTGRES_DB=mydb
    ports:
      - "5432:5432"

  api:
    build: ./api
    environment:
      - DATABASE_URL=postgres://postgres:postgres@postgres:5432/mydb
    depends_on:
      - postgres

  web:
    build: ./web
    environment:
      - API_URL=http://api:8080
    depends_on:
      - api
```

</TabItem>
<TabItem label='C#'>

```csharp title="AppHost.cs" showLineNumbers
var builder = DistributedApplication.CreateBuilder(args);

var db = builder.AddPostgres("postgres")
    .AddDatabase("mydb");

var api = builder.AddProject<Projects.Api>("api")
    .WithReference(db);

var web = builder.AddProject<Projects.Web>("web")
    .WithExternalHttpEndpoints()
    .WithReference(api)
    .WaitFor(api);

builder.Build().Run();
```

</TabItem>
<TabItem label='TypeScript'>

```typescript title="apphost.ts" showLineNumbers
import { createBuilder } from './.modules/aspire.js';

const builder = await createBuilder();

const db = (await builder.addPostgres('postgres')).addDatabase('mydb');

const api = await builder
    .addNodeApp('api', './api', 'server.js')
    .withNpm()
    .withReference(db);

await builder
    .addViteApp('web', './web')
    .withExternalHttpEndpoints()
    .withReference(api)
    .waitFor(api);

await builder.build().run();
```
