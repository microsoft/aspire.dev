The official Aspire extension for [Visual Studio Code](https://code.visualstudio.com/) is the easiest way to debug a polyglot Aspire app. Aspire apps usually span several languages — C#, TypeScript, Python, browser apps, Azure Functions — and the extension takes a single <Kbd windows='F5' /> and turns it into a coordinated debug session that builds your AppHost, starts every resource (services, containers, databases), attaches the right debugger to each language, and opens the dashboard. While your app runs, the Aspire view in the Side Bar shows live resource state, CodeLens paints state directly into your AppHost source, and the Command Palette covers everything else: scaffolding new projects, adding integrations, and executing pipeline steps without leaving the editor.

:::note[Prerequisites]
A recent version of [Visual Studio Code](https://code.visualstudio.com/) and the **Aspire CLI** — install the CLI via the **Aspire: Install Aspire CLI (stable)** command from the Command Palette, or follow the [installation guide](/get-started/install-cli/).
:::

<Steps>

## Getting started

Open your Aspire project in VS Code, or create one with **Aspire: New Aspire project** from the Command Palette (<Kbd windows='Ctrl+Shift+P' mac='Cmd+Shift+P' />). Run **Aspire: Configure launch.json file** to set up the debug configuration, then press <Kbd windows='F5' />.

<ThemeImage
  dark={createProject}
  light={createProjectLight}
  alt="VS Code showing the Aspire new project command with template options."
/>

The extension discovers your AppHost from the workspace using `aspire.config.json`, so both project-based C# AppHosts (`*.csproj`) and TypeScript AppHosts (`apphost.ts`) are picked up automatically. The Aspire CLI is detected at its default install location even if it's not on PATH; if `code` isn't on PATH the extension falls back to `code-insiders`.

:::tip[Prefer a guided tour?]
Open **Help > Get Started > Get started with Aspire** for the built-in walkthrough — it covers installing the CLI (with clickable terminal links that run the install commands for you), creating a project, running the app, exploring the dashboard, and next steps, all without leaving VS Code.
:::

## Live editor integration

When an AppHost is running, the extension paints live state directly into your editor.

{/* TODO: <ThemeImage dark={...} light={...} alt="An AppHost source file with CodeLens above each resource definition showing state and Start/Stop/Restart/View Logs actions, and colored gutter dots in the margin." /> — file: src/frontend/src/assets/get-started/code-extension-codelens{,-light}.png */}

**CodeLens** appears above each resource definition in C# (`DistributedApplication.CreateBuilder`) and TypeScript (`createBuilder()`) AppHost files. The CodeLens shows the resource's current state and health (Running, Starting, Stopped, Error) and exposes one-click actions: Start, Stop, Restart, and View Logs, plus any custom commands the resource publishes from the runtime.

**Gutter decorations** color-code each resource definition in the gutter — green for running, blue for starting, red for errors, gray for stopped — so you get an at-a-glance health view of the whole AppHost without leaving the source.

## The Aspire view

The extension adds an **Aspire** view to the Side Bar — click the Aspire icon in the Activity Bar to open it. The view lists every running AppHost and its resources, and state changes appear in real time with no polling. **Workspace** mode shows resources for the AppHost in your current workspace; toggle to **Global** mode in the view header to see every AppHost running on your machine.

{/* TODO: <ThemeImage dark={...} light={...} alt="The Aspire view showing a tree of resources with health-aware icons and a hover tooltip listing endpoint URLs." /> — file: src/frontend/src/assets/get-started/code-extension-sidebar{,-light}.png */}

Each resource shows a health-aware icon (green for healthy, red for errors, amber for degraded, spinner for starting) and a rich markdown tooltip with its type, state, and clickable endpoint URLs. Right-click a resource to start, stop, or restart it, view its logs, run resource-specific commands, or open the dashboard. Right-click an endpoint URL to open it in your browser, copy it, and so on.

## Run, debug, and deploy

The extension auto-detects your AppHost from the workspace, so a minimal `.vscode/launch.json` works for any AppHost language — C#, file-based C# (`apphost.cs`), TypeScript (`apphost.ts`), and so on:

```json title=".vscode/launch.json — launch"
{
  "type": "aspire",
  "request": "launch",
  "name": "Aspire: Launch AppHost",
  "program": "${workspaceFolder}"
}
```

Or run **Aspire: Configure launch.json file** from the Command Palette to generate one automatically. If you have multiple AppHosts in the workspace, point `program` at a specific one — for example, `${workspaceFolder}/MyAppHost/MyAppHost.csproj` (project-based C#), `${workspaceFolder}/MyAppHost/apphost.cs` (file-based C#), or `${workspaceFolder}/apphost.ts` (TypeScript).

When you press <Kbd windows='F5' />, the extension builds the AppHost, starts all resources in the right order, attaches debuggers based on each service's language, and opens the dashboard. AppHost exit messages are surfaced in the debug console so failures are easy to spot. You can also right-click an AppHost file (`*.csproj`, `apphost.cs`, or `apphost.ts`) in the Explorer and pick **Run Aspire apphost** or **Debug Aspire apphost**.

The `command` property in the launch config does more than just running. Set it to `deploy` to push to your defined deployment targets, `publish` to generate deployment artifacts (manifests, Bicep files, and the like), or `do` to execute a specific pipeline step (set `step` to the step name). All three flows can be debugged from VS Code — you can step into your custom publishers and deployment steps the same way you step into application code.

```json title=".vscode/launch.json — deploy"
{
  "type": "aspire",
  "request": "launch",
  "name": "Aspire: Deploy AppHost",
  "program": "${workspaceFolder}",
  "command": "deploy"
}
```

The `debuggers` property lets you pass debug configuration specific to a class of resource. The extension recognizes a handful of keys: `apphost` for the AppHost itself, `project` for C#/.NET projects, `node` for Node.js resources, `python` for Python resources, `browser` for browser apps, and `azure-functions` for Azure Functions. Anything you set under one of those keys is merged into the underlying VS Code debug configuration, so you can pass any property the corresponding debug adapter understands. Mix and match for polyglot apps:

```json title=".vscode/launch.json — debugger settings"
{
  "type": "aspire",
  "request": "launch",
  "name": "Aspire: Launch AppHost",
  "program": "${workspaceFolder}",
  "debuggers": {
    "apphost": {
      "stopAtEntry": true
    },
    "project": {
      "console": "integratedTerminal",
      "logging": { "moduleLoad": false }
    },
    "node": {
      "skipFiles": ["<node_internals>/**"]
    },
    "python": {
      "justMyCode": false
    }
  }
}
```

## Language coverage

The extension detects each resource's language and attaches the right debugger. The required VS Code extension depends on the language:

| Language | Debugger | Required VS Code extension |
|---|---|---|
| Node.js | `js-debug` | None (built into VS Code) |
| Browser apps | `js-debug` | None (built into VS Code) |
| Python | `debugpy` | [Python](https://marketplace.visualstudio.com/items?itemName=ms-python.python) |
| C# / .NET | `coreclr` | [C# Dev Kit](https://marketplace.visualstudio.com/items?itemName=ms-dotnettools.csdevkit) (recommended) or [C#](https://marketplace.visualstudio.com/items?itemName=ms-dotnettools.csharp) |
| Azure Functions | Per language | [Azure Functions](https://marketplace.visualstudio.com/items?itemName=ms-azuretools.vscode-azurefunctions) plus the language extension for your function app (C#, JavaScript/TypeScript, or Python) |

TypeScript AppHosts (`createBuilder()`) get the same editor integration as C# AppHosts — CodeLens, gutter decorations, the Aspire view, and debugging.

## MCP server integration

The extension can register an Aspire [MCP](https://modelcontextprotocol.io/) server with VS Code. This lets AI tools — GitHub Copilot included — see your running app's resources, endpoints, and configuration for better context when helping you write code.

Turn it on by setting `aspire.registerMcpServerInWorkspace` to `true`. When enabled, the extension auto-registers the MCP server definition via the Aspire CLI whenever a workspace is open and the CLI is available.

## Reference

### Settings

Configure the extension under **Settings > Aspire**, or jump there with **Aspire: Extension settings**. The settings you'll touch most often:

| Setting | Default | Description |
|---------|---------|-------------|
| `aspire.aspireCliExecutablePath` | `""` | Path to the Aspire CLI. Leave empty to let the extension auto-detect (PATH first, then default install location). |
| `aspire.dashboardBrowser` | `openExternalBrowser` | Which browser opens the dashboard — system default, or Chrome/Edge/Firefox as a debug session. |
| `aspire.enableAspireDashboardAutoLaunch` | `true` | Open the dashboard automatically when you start debugging. |
| `aspire.registerMcpServerInWorkspace` | `false` | Register the Aspire MCP server (see [MCP server integration](#mcp-server-integration)). |

There are more settings for verbose logging, startup prompts, and polling intervals — open **Aspire: Extension settings** to see them all. The extension also provides IntelliSense and validation when editing `.aspire/settings.json` (workspace-level) and `~/.aspire/globalsettings.json` (user-level).

### Commands

<details>
<summary>All commands available from the Command Palette under the **Aspire** category</summary>

**Project lifecycle.** **New Aspire project** scaffolds a new AppHost or starter app from a template. **Initialize Aspire in an existing codebase** wires Aspire into a project you already have. **Add an integration** picks a hosting integration (`Aspire.Hosting.*`) and adds it to the AppHost; **Update integrations** keeps those packages and the Aspire SDK current.

**Run, debug, and deploy.** **Configure launch.json file** drops the Aspire debug entry into your workspace. **Publish deployment artifacts**, **Deploy app**, and **Execute pipeline step** map directly to `aspire publish`, `aspire deploy`, and `aspire do` — handy when you want to invoke a single step without setting up a launch config.

**Settings and CLI management.** **Extension settings** opens the Aspire section of VS Code settings. **Open local Aspire settings** edits `.aspire/settings.json` for the current workspace; **Open global Aspire settings** edits `~/.aspire/globalsettings.json`. **Open Aspire terminal** spawns a terminal with the Aspire CLI on PATH. **Install Aspire CLI (stable)**, **Install Aspire CLI (daily)**, **Update Aspire CLI**, and **Verify Aspire CLI installation** cover installing and maintaining the CLI without leaving the editor.

</details>
