The official Aspire extension for [Visual Studio Code](https://code.visualstudio.com/) makes debugging polyglot Aspire apps a one-keystroke affair. Press <Kbd windows='F5' /> and the extension builds your AppHost, starts every resource, attaches the right debugger to each language — C#, TypeScript, Python, browser apps, Azure Functions — and opens the dashboard. Live state surfaces inline as CodeLens and gutter icons, and the Aspire view in the Side Bar tracks every running AppHost in real time.

:::note[Prerequisites]
A recent version of [Visual Studio Code](https://code.visualstudio.com/) and the **Aspire CLI** — install the CLI via the **Aspire: Install Aspire CLI (stable)** command from the Command Palette, or follow the [installation guide](/get-started/install-cli/).
:::

## Install the extension

<Steps>

1. Open VS Code.
1. Open the **Extensions** view (<Kbd windows='Ctrl+Shift+X' mac='Cmd+Shift+X' />).
1. Search for **Aspire** and install the extension published by **Microsoft**, or install it directly from the [VS Code Marketplace](https://aka.ms/aspire/vscode).

</Steps>

## Getting started

Open your Aspire project in VS Code, or create one with **Aspire: New Aspire project** from the Command Palette (<Kbd windows='Ctrl+Shift+P' mac='Cmd+Shift+P' />). Run **Aspire: Configure launch.json file** to set up the debug configuration, then press <Kbd windows='F5' />.

<ThemeImage
  dark={createProject}
  light={createProjectLight}
  alt="VS Code showing the Aspire new project command with template options."
/>

The extension discovers your AppHost from `aspire.config.json`, so both project-based C# AppHosts (`*.csproj`) and TypeScript AppHosts (`apphost.ts`) work without extra configuration.

:::tip[Prefer a guided tour?]
**Help > Get Started > Get started with Aspire** opens a built-in walkthrough covering CLI install, project creation, running, and exploring the dashboard — all without leaving VS Code.
:::

## Live editor integration

When an AppHost is running, the extension paints live state directly into your editor.

{/* TODO: <ThemeImage dark={...} light={...} alt="An AppHost source file with CodeLens above each resource definition showing state and Start/Stop/Restart/View Logs actions, and colored gutter dots in the margin." /> — file: src/frontend/src/assets/get-started/code-extension-codelens{,-light}.png */}

**CodeLens** appears above each resource definition in C# (`DistributedApplication.CreateBuilder`) and TypeScript (`createBuilder()`) AppHosts. It shows the current state and health (for example, `Running`, `Stopped (Exit code: 0)`, or `Running - (Unhealthy 0/1)`) and exposes one-click Start, Stop, Restart, and View Logs actions — plus any custom commands the resource publishes. Hover for per-check health details.

**Gutter decorations** mark each resource definition with a state-coded shape that won't clash with breakpoint indicators:

| Icon | Meaning |
|------|---------|
| ✓ (green checkmark) | Running and healthy, or completed successfully (exit code 0) |
| ✓ (pale green checkmark) | Completed — resource exited cleanly |
| ⚠ (yellow triangle) | Running but unhealthy (health checks failing) |
| ✕ (red X) | Error — failed, exited with non-zero exit code, or unknown |
| ⌛ (hourglass) | Starting, stopping, or waiting |
| ○ (hollow circle) | Not yet started |

## The Aspire view

The extension adds an **Aspire** view to the Side Bar. It lists every running AppHost and its resources, with state changes appearing in real time. **Workspace** mode shows the AppHost in your current workspace; toggle to **Global** in the view header to see every AppHost on your machine.

{/* TODO: <ThemeImage dark={...} light={...} alt="The Aspire view showing a tree of resources with health-aware icons and a hover tooltip listing endpoint URLs." /> — file: src/frontend/src/assets/get-started/code-extension-sidebar{,-light}.png */}

Each resource shows its type, state, health summary, and exit code (for example, `Project · Running · Health: 1/2`), plus a health-aware icon and a markdown tooltip listing endpoint URLs. Resources with health checks expose an expandable **Health Checks** group; use **Expand all** on the AppHost item to open everything at once.

Right-click a resource to start, stop, or restart it, view its logs, run resource-specific commands, or open the dashboard. Right-click an endpoint URL to open or copy it.

## Run, debug, and deploy

{/* TODO: <ThemeImage dark={...} light={...} alt="VS Code Run & Debug view with multiple debug sessions attached — AppHost, a .NET project, a Node service, and a browser app — alongside the dashboard." /> — file: src/frontend/src/assets/get-started/code-extension-debug{,-light}.png */}

The extension auto-detects your AppHost, so a minimal `.vscode/launch.json` works for any AppHost language:

```json title=".vscode/launch.json — launch"
{
  "type": "aspire",
  "request": "launch",
  "name": "Aspire: Launch AppHost",
  "program": "${workspaceFolder}"
}
```

Or run **Aspire: Configure launch.json file** to generate one. With multiple AppHosts in the workspace, point `program` at a specific file — a `.csproj`, `apphost.cs` (file-based C#), or `apphost.ts` (TypeScript).

Press <Kbd windows='F5' /> to build, start every resource in order, attach debuggers, and open the dashboard — AppHost exit messages surface in the debug console. You can also right-click an AppHost file in the Explorer and pick **Run Aspire apphost** or **Debug Aspire apphost**.

Set `command` to `deploy` (push to deployment targets), `publish` (generate manifests, Bicep, etc.), or `do` (run a specific pipeline step — set `step` to its name). All three are debuggable; you can step into custom publishers and deployment steps just like application code.

```json title=".vscode/launch.json — deploy"
{
  "type": "aspire",
  "request": "launch",
  "name": "Aspire: Deploy AppHost",
  "program": "${workspaceFolder}",
  "command": "deploy"
}
```

The `debuggers` property forwards configuration to specific debuggers. Recognized keys: `apphost`, `project` (C#/.NET), `node`, `python`, `browser`, and `azure-functions`. Anything you set under a key is merged into the corresponding VS Code debug configuration:

```json title=".vscode/launch.json — debugger settings"
{
  "type": "aspire",
  "request": "launch",
  "name": "Aspire: Launch AppHost",
  "program": "${workspaceFolder}",
  "debuggers": {
    "apphost": { "stopAtEntry": true },
    "project": { "console": "integratedTerminal" },
    "node": { "skipFiles": ["<node_internals>/**"] },
    "python": { "justMyCode": false }
  }
}
```

## Language coverage

The extension detects each resource's language and attaches the right debugger:

| Language | Debugger | Required VS Code extension |
|---|---|---|
| Node.js / Browser apps | `js-debug` | None (built into VS Code) |
| Python | `debugpy` | [Python](https://marketplace.visualstudio.com/items?itemName=ms-python.python) |
| C# / .NET | `coreclr` | [C# Dev Kit](https://marketplace.visualstudio.com/items?itemName=ms-dotnettools.csdevkit) (recommended) or [C#](https://marketplace.visualstudio.com/items?itemName=ms-dotnettools.csharp) |
| Azure Functions | Per language | [Azure Functions](https://marketplace.visualstudio.com/items?itemName=ms-azuretools.vscode-azurefunctions) plus the language extension for your function app (C#, JavaScript/TypeScript, or Python) |

## Commands

<Expand summary="All commands available from the Command Palette under the Aspire category">

**Project lifecycle.** **New Aspire project** scaffolds an AppHost or starter app from a template. **Initialize Aspire in an existing codebase** wires Aspire into a project you already have. **Add an integration** picks a hosting integration (`Aspire.Hosting.*`); **Update integrations** keeps those packages and the Aspire SDK current.

**Run, debug, and deploy.** **Configure launch.json file** drops the Aspire debug entry into your workspace. **Publish deployment artifacts**, **Deploy app**, and **Execute pipeline step** map to `aspire publish`, `aspire deploy`, and `aspire do`.

**Settings and CLI.** **Extension settings**, **Open local Aspire settings** (`.aspire/settings.json`), and **Open global Aspire settings** (`~/.aspire/globalsettings.json`) cover configuration. **Restore** runs `aspire restore` to sync integration packages. **Open Aspire terminal** spawns a terminal with the CLI on PATH. **Install Aspire CLI (stable / daily)**, **Update Aspire CLI**, and **Verify Aspire CLI installation** manage the CLI from the editor.

</Expand>
