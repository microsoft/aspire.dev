## Overview

HTTPS/TLS secured endpoints are essential for protecting data in transit, ensuring authenticity, and maintaining trust between services. Even in development environments, properly configured HTTPS helps identify and resolve certificate-related issues before they reach production.

However, working with self-signed certificates during local development presents challenges:

- **Certificate trust**: Browsers and HTTP clients reject self-signed certificates by default
- **Certificate distribution**: Sharing certificates across multiple services and containers is error-prone
- **Platform differences**: Certificate management varies significantly between Windows, macOS, and Linux
- **Container complexity**: Configuring certificate trust inside containers requires understanding each container's certificate bundle system

Aspire provides several APIs to simplify HTTPS/TLS certificate configuration in development environments, making it straightforward to:

- Use the ASP.NET Core developer certificate across all resources
- Provide custom certificates for specific services
- Configure certificate trust between resources
- Handle certificate distribution to containers automatically

:::note
The certificate configuration APIs described in this article apply to local development scenarios (run mode). In publish mode, resources use their default certificate configuration appropriate for production environments.
:::

## Using a .NET developer certificate

The simplest approach for enabling HTTPS/TLS in development with Aspire is to use a self-signed certificate generated by the .NET SDK and valid for local development. The Aspire CLI automates the process of generating a developer certificate if you haven't already done so and Aspire can automatically configure resources to trust this certificate and use it for HTTPS endpoints. This approach works whether your service resources are .NET, Python, or JavaScript based.

### Securing endpoints with the developer certificate

While ASP.NET Core projects will automatically use the developer certificate, Aspire extends this support to severeal other resource types including Python Uvicorn based projects and YARP reverse proxy containers. The currently supported resources include:

- ASP.NET Core projects
- Python Uvicorn projects
- YARP reverse proxy containers
- Redis containers
- Keycloak containers

An opt-in implementation is provided for Vite resources and can be enabled using:

```csharp title="C# — Program.cs"
var builder = DistributedApplication.CreateBuilder(args);
builder.AddViteApp("frontend", "./frontend")
    .WithHttpsDeveloperCertificate(); // Configure Vite to use HTTPS with the developer certificate
```

::: note
The APIs for configuring HTTPS endpoints are currently experimental and may change in future releases.
:::

#### Per-resource HTTPS certificate configuration

You can disable HTTPS certificate configuration for a specific resource using `WithoutHttpsCertificate`:

```csharp title="C# — Program.cs"
var redis = builder.AddRedis("cache")
    .WithoutHttpsCertificate(); // Disable HTTPS for this Redis container
```

Additionally, you can provide a custom X.509 certificate to use for your HTTPS/TLS endpoints instead of using the developer certificate:

```csharp title="C# — Program.cs"
using System.Security.Cryptography.X509Certificates;
var certificate = new X509Certificate2("./path/to/certificate.pfx");
var redis = builder.AddRedis("cache")
    .WithHttpsCertificate(certificate); // Use a custom certificate for this Redis container
```

You can also specify a password parameter to encrypt a certificate's private key:

```csharp title="C# — Program.cs"
using System.Security.Cryptography.X509Certificates;

var certificate = new X509Certificate2("./path/to/certificate.pfx");
var certPassword = builder.AddParameter("cert-password", secret: true);

var container = builder.AddContainer("my-service", "my-image")
    .WithHttpsDeveloperCertificate(certPassword);

var redis = builder.AddRedis("cache")
    .WithHttpsCertificate(certificate, certPassword);
```

::: note
Not all resource types support password-protected certificates. In such cases, HTTPS endpoints may not work as expected.
:::

#### Default behavior for all resources

You can disable the default behavior of configuring compatible resources to use the developer certificate by setting `DeveloperCertificateDefaultHttpsTerminationEnabled` to `false` when creating the application builder:

```csharp title="C# — Program.cs"
var builder = DistributedApplication.CreateBuilder(new DistributedApplicationOptions
{
    Args = args,
    DeveloperCertificateDefaultHttpsTerminationEnabled = false, // Disable by default, all resources will need to have https explicitly enabled
});
```

Alternatively you can set an environment variable to control this behavior:

<Tabs syncKey="terminal">
    <TabItem label="Bash" icon="seti:shell">
        ```bash
        ASPIRE_DEVELOPER_CERTIFICATE_DEFAULT_HTTPS_TERMINATION=false
        ```
    </TabItem>
    <TabItem label="PowerShell" icon="seti:powershell">
        ```powershell
        $env:ASPIRE_DEVELOPER_CERTIFICATE_DEFAULT_HTTPS_TERMINATION="false"
        ```
    </TabItem>
</Tabs>

If you want to make this the global default for Aspire projects, you can add this environment variable in your user or system environment.

### Control developer certificate trust

Use `WithDeveloperCertificateTrust` to control whether a resource should trust the developer certificate as a certificate authority:

```csharp title="C# — Program.cs"
// Disable trust for developer certificate on a specific resource
var container = builder.AddContainer("my-service", "my-service:latest")
    .WithDeveloperCertificateTrust(false);
```

This is useful when you've disabled automatic trust globally but want to enable it for specific resources:

```csharp title="C# — Program.cs"
var builder = DistributedApplication.CreateBuilder(new DistributedApplicationOptions
{
    Args = args,
    TrustDeveloperCertificate = false,
});

// Explicitly enable for this project
var project = builder.AddProject<Projects.MyService>("my-service")
    .WithDeveloperCertificateTrust(true);
```

## Advanced certificate configuration

For resources that require custom certificate configuration (such as passing certificate paths via specific command-line arguments or environment variables), use the configuration callback APIs.

### Configure HTTPS certificate paths

Use `WithHttpsCertificateConfiguration` to access certificate paths and configure how a resource uses them:

```csharp title="C# — Program.cs"
var container = builder.AddContainer("my-service", "my-image")
    .WithHttpsDeveloperCertificate()
    .WithHttpsCertificateConfiguration(ctx =>
    {
        // Add arguments pointing to the certificate files
        ctx.Arguments.Add("--cert-path");
        ctx.Arguments.Add(ctx.CertificatePath);
        ctx.Arguments.Add("--key-path");
        ctx.Arguments.Add(ctx.KeyPath);

        // Set environment variables if the service expects them
        ctx.EnvironmentVariables["TLS_CERT_FILE"] = ctx.CertificatePath;
        ctx.EnvironmentVariables["TLS_KEY_FILE"] = ctx.KeyPath;

        return Task.CompletedTask;
    });
```

The callback context provides:

- `CertificatePath`: Path to the certificate file (PEM format)
- `KeyPath`: Path to the private key file (PEM format)
- `PfxPath`: Path to the certificate in PKCS#12 (.pfx) format
- `Password`: The password parameter, if provided
- `Arguments`: Collection to add command-line arguments
- `EnvironmentVariables`: Collection to set environment variables

Here's an example from the Redis integration showing how it configures TLS:

```csharp title="C# — RedisBuilderExtensions.cs"
redis.WithHttpsCertificateConfiguration(ctx =>
{
    ctx.Arguments.Add("--tls-cert-file");
    ctx.Arguments.Add(ctx.CertificatePath);
    ctx.Arguments.Add("--tls-key-file");
    ctx.Arguments.Add(ctx.KeyPath);

    return Task.CompletedTask;
});
```

## Configuring certificate trust

When resources need to trust custom certificate authorities (such as self-signed certificates from other services), Aspire provides APIs to configure certificate trust.

### Add custom certificate authorities

Use `AddCertificateAuthorityCollection` to create a collection of trusted certificates, then associate it with resources using `WithCertificateAuthorityCollection`:

```csharp title="C# — Program.cs"
var builder = DistributedApplication.CreateBuilder(args);

// Create a collection of trusted CAs
var customCAs = builder.AddCertificateAuthorityCollection("custom-cas")
    .WithCertificatesFromFile("../path/to/ca-bundle.pem");

// Configure a container to trust these CAs
var container = builder.AddContainer("my-service", "my-service:latest")
    .WithCertificateAuthorityCollection(customCAs);
```

You can add certificates from multiple sources:

```csharp title="C# — Program.cs"
using System.Security.Cryptography.X509Certificates;

var customCAs = builder.AddCertificateAuthorityCollection("custom-cas")
    // Add from a PEM file
    .WithCertificatesFromFile("../ca-bundle.pem")
    // Add individual certificate
    .WithCertificate(new X509Certificate2("../another-ca.pem"))
    // Add from certificate store
    .WithCertificatesFromStore(StoreName.Root, StoreLocation.LocalMachine);
```

### Configure certificate trust scope

The `WithCertificateTrustScope` method controls how custom certificate authorities are applied to a resource. This is particularly important for containers that have specific certificate bundle configurations.

```csharp title="C# — Program.cs"
var customCAs = builder.AddCertificateAuthorityCollection("custom-cas")
    .WithCertificatesFromFile("../ca-bundle.pem");

var container = builder.AddContainer("my-service", "my-service:latest")
    .WithCertificateAuthorityCollection(customCAs)
    .WithCertificateTrustScope(CertificateTrustScope.Override);
```

Available scopes:

- **`Append`** (default): Adds custom certificate authorities to the resource's default trusted CAs. This is the safest option as it maintains trust for public CAs while adding your custom certificates.

  ```csharp title="C# — Program.cs"
  .WithCertificateTrustScope(CertificateTrustScope.Append)
  ```

- **`Override`**: Replaces the default trusted CAs with only the provided custom certificates. Use this when you want complete control over which certificates are trusted.

  ```csharp title="C# — Program.cs"
  .WithCertificateTrustScope(CertificateTrustScope.Override)
  ```

- **`System`**: Configures the resource to trust both the default system certificate authorities and the custom certificates. Useful for resources that don't support appending to their default trusted CAs but allow overriding the entire trust store.

  ```csharp title="C# — Program.cs"
  .WithCertificateTrustScope(CertificateTrustScope.System)
  ```

- **`None`**: Disables all custom certificate authority configuration, using only the resource's default certificate trust behavior.

  ```csharp title="C# — Program.cs"
  .WithCertificateTrustScope(CertificateTrustScope.None)
  ```

### Configure certificate trust paths

For advanced scenarios where you need to control exactly how certificate bundles are passed to a resource, use `WithCertificateTrustConfiguration`:

```csharp title="C# — Program.cs"
var container = builder.AddContainer("my-service", "my-service:latest")
    .WithCertificateTrustConfiguration(ctx =>
    {
        // Configure environment variable for certificate bundle
        if (ctx.Scope != CertificateTrustScope.Append)
        {
            ctx.EnvironmentVariables["SSL_CERT_FILE"] = ctx.CertificateBundlePath;
        }

        // Add directory for additional certificates
        ctx.EnvironmentVariables["SSL_CERT_DIR"] = ctx.CertificateDirectoriesPath;

        return Task.CompletedTask;
    });
```

The callback context provides:

- `CertificateBundlePath`: Path to a bundle file containing all custom CAs
- `CertificateDirectoriesPath`: Path to a directory containing individual CA certificate files
- `Scope`: The configured `CertificateTrustScope` for the resource
- `Arguments`: Collection to add command-line arguments
- `EnvironmentVariables`: Collection to set environment variables

Here's an example from the Redis integration:

```csharp title="C# — RedisBuilderExtensions.cs"
redis.WithCertificateTrustConfiguration(ctx =>
{
    ctx.Arguments.Add("--tls-ca-cert-file");
    ctx.Arguments.Add(ctx.CertificateBundlePath);

    return Task.CompletedTask;
});
```

## Common scenarios

### API gateway with custom certificate

```csharp title="C# — Program.cs"
var certificate = new X509Certificate2("gateway-cert.pfx", "password");

var gateway = builder.AddContainer("api-gateway", "gateway-image")
    .WithHttpsCertificate(certificate)
    .WithHttpsCertificateConfiguration(ctx =>
    {
        ctx.EnvironmentVariables["TLS_CERT"] = ctx.PfxPath;
        ctx.EnvironmentVariables["TLS_PASSWORD"] = ctx.Password?.Value ?? "";
        return Task.CompletedTask;
    });
```

### Service trusting custom CA

```csharp title="C# — Program.cs"
// Create a CA collection with your organization's root CA
var orgCAs = builder.AddCertificateAuthorityCollection("org-cas")
    .WithCertificatesFromFile("org-root-ca.pem");

// Backend service uses a cert signed by the org CA
var backendCert = new X509Certificate2("backend-cert.pfx");
var backend = builder.AddProject<Projects.Backend>("backend")
    .WithHttpsCertificate(backendCert)
    .WithCertificateAuthorityCollection(orgCAs);

// Frontend trusts the org CA to verify backend's certificate
var frontend = builder.AddProject<Projects.Frontend>("frontend")
    .WithReference(backend)
    .WithCertificateAuthorityCollection(orgCAs);
```

### Microservices with mutual TLS

```csharp title="C# — Program.cs"
// Shared CA for all services
var sharedCA = builder.AddCertificateAuthorityCollection("shared-ca")
    .WithCertificatesFromFile("service-ca.pem");

// Each service gets its own certificate and trusts the shared CA
var serviceCert = new X509Certificate2("service1-cert.pfx");
var service1 = builder.AddContainer("service1", "service1-image")
    .WithHttpsCertificate(serviceCert)
    .WithCertificateAuthorityCollection(sharedCA);

var service2Cert = new X509Certificate2("service2-cert.pfx");
var service2 = builder.AddContainer("service2", "service2-image")
    .WithHttpsCertificate(service2Cert)
    .WithCertificateAuthorityCollection(sharedCA);
```
